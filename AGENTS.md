# 🍁 메이플스토리 감성 도트 스케줄러 — 통합 기획·설계 문서

---

## 1. 기획 및 요구사항 정의

### 1-1. 사용자 경험(UX) 전략 — '도트 감성' 핵심 요소

일반 캘린더와의 결정적 차이는 **"일정을 처리한다"가 아니라 "퀘스트를 클리어한다"**는 감각입니다. 이를 위한 필수 요소:

- **시각적 요소:** 8~16px 그리드 기반 픽셀 아트, 계단식 안티앨리어싱 없는 하드 엣지, 메이플 특유의 따뜻한 파스텔+원색 대비, 캐릭터 도트 스프라이트(idle 애니메이션 2~4프레임 루프), 말풍선/퀘스트 알림창의 픽셀 테두리(9-slice 방식).
- **상호작용 피드백:** 버튼 호버 시 1px 떨림 또는 색 반전, 클릭 시 "딸깍" 픽셀 사운드, 일정 완료 시 캐릭터 머리 위 `+EXP` 도트 숫자가 튀어오르는 모션, 레벨업 시 화면 전체 플래시 + 황금 파티클.
- **사운드 디자인:** 메이플 BGM을 연상시키는 칩튠(chiptune) 루프, 완료 시 코인 사운드, 레벨업 시 팡파레. *단, 자동재생 금지 — 사용자 토글 필수.*
- **마이크로 인터랙션 원칙:** 모든 행동에 0.1~0.3초 내 즉각 반응(쾌감의 핵심), 빈 화면 없이 항상 캐릭터가 '살아있는' 느낌(대기 모션, 랜덤 이모트).

### 1-2. 게이미피케이션 보상 체계

| 보상 유형 | 획득 조건 | 지속 접속 유도 메커니즘 |
|---|---|---|
| **경험치(EXP)** | 일정 완료 시 난이도별 차등 지급 | 다음 레벨까지의 게이지가 보이는 '근접 보상' 심리 |
| **레벨업** | EXP 누적 | 일정 칸 수 증가, 신규 테마 해금 |
| **배지(업적)** | "7일 연속 완료", "하루 5개 클리어" 등 | 컬렉션 욕구 자극 |
| **칭호** | 누적 통계 달성("불꽃의 노력가") | 캐릭터 위 상시 노출 → 자기표현 |
| **꾸미기 아이템** | 레벨/코인/이벤트 | 캐릭터·배경·캘린더 스킨 커스터마이징 |
| **코인(메소)** | 일일 출석·완료 | 상점 구매 재화 → 매일 접속 동기 |
| **연속 출석(스트릭)** | 매일 접속 | 스트릭 깨짐 방지 심리(손실 회피) |

**핵심 설계 원칙:** ① 첫 7일 안에 최소 3회 레벨업 경험(온보딩 도파민), ② 일일 퀘스트로 '오늘 할 이유' 제공, ③ 보상은 즉시성(완료 즉시)과 누적성(주간/월간) 병행.

### 1-3. 친구 기능 — 협동 & 경쟁

- **협동 퀘스트(파티 플레이):** 친구와 공동 목표 설정("우리 둘이 이번 주 운동 10회"), 합산 게이지 달성 시 양쪽 모두 보너스 EXP. 한쪽이 부진하면 알림으로 독려("○○님이 기다리고 있어요!").
- **경쟁 요소:** 주간 EXP 랭킹 리더보드(친구 한정), 길드(소규모 그룹) 단위 누적 점수 경쟁, '완료율 대결' 1:1 매치.
- **상호작용:** 친구 일정에 '응원 스티커' 남기기, 완료 시 자동 축하 알림, 친구 캐릭터 방문(마이룸 개념).
- **주의:** 경쟁이 압박이 되지 않도록 **'협동 우선, 경쟁은 선택'** 구조 — 경쟁 기능은 옵트인(opt-in)으로.

### 1-4. 핵심 차별화 전략 — '플레이적 경험'

> **"할 일을 적는 순간, 모험이 시작된다."**

- **생산성 앱과의 결정적 차이:** Todoist·구글캘린더는 '관리 도구', 이 앱은 **'나의 성장을 시각화하는 RPG'**. 일정 = 퀘스트, 나 = 성장하는 캐릭터.
- **감정적 차별화:** 30~40대 메이플 향수 세대를 타깃 — "그때 그 감성으로 오늘을 산다"는 정서적 후킹.
- **유지율 무기:** 일정 완료의 '의무감'을 '성취 쾌감'으로 전환. 빈 캘린더가 아니라 "내가 키운 캐릭터와 누적된 배지"를 보여줌 → 떠나기 아까운 자산화.

## 2. UI/UX 설계 및 와이어프레임

- **상단 HUD:** 게임의 캐릭터 정보창 차용. 경험치 바는 항상 노출(목표 가시성).
- **중앙 캘린더:** 셀 자체가 픽셀 카드. 완료한 날은 도트 도장이 찍히고, 미완료는 흐린 톤.
- **하단 퀘스트 로그:** 체크박스 클릭 = 퀘스트 완료 연출 트리거.

### 2-2. 사용자 흐름(User Flow)

```
[랜딩] 
  → "모험 시작하기" (구글 로그인)
  → [신규] 캐릭터 생성: 외형/직업 선택 + 닉네임 → 튜토리얼 퀘스트
  → [기존] 캐릭터 선택 → 대시보드 진입
       ↓
  [일정 등록] ＋버튼 → 제목/날짜/반복여부/난이도 입력 → "퀘스트 수락!"
       ↓
  [일정 수행] 체크박스 클릭 → 완료 애니메이션 + EXP 획득 연출
       ↓
  [경험치 누적] → 게이지 충전 → (임계치 도달) 레벨업 풀스크린 연출
       ↓
  [보상 수령] 코인/아이템 지급 → 상점·꾸미기에서 사용
```

온보딩은 **"첫 일정 등록 → 즉시 완료 → 레벨업"**까지 3분 안에 경험시키는 것이 핵심(첫 도파민 빠르게).

### 2-3. 디자인 시스템

- **폰트:** 한글 픽셀 폰트 **둥근모꼴(DungGeunMo)** 또는 **갈무리11**, 영문/숫자는 **Press Start 2P** 또는 **Galmuri**. `image-rendering: pixelated` 필수 적용.
- **컬러 팔레트(예시):**
  - 배경: `#F4E4C1`(양피지 베이지), `#2B2B3D`(다크모드)
  - 포인트: `#FF8C42`(메이플 오렌지), `#FFD23F`(EXP 골드)
  - 성공/완료: `#6BCB77`, 경고: `#FF5252`
  - UI 테두리: `#3A2E2A`(짙은 갈색 1~2px)
- **도트 이펙트 가이드:**
  - 완료 시: 체크 → 항목 회색 페이드 + 머리 위 `+EXP` 숫자가 위로 튀며 사라짐(0.6s, ease-out)
  - 레벨업: 화면 황금 플래시(0.2s) → 캐릭터 점프 + 파티클 8방향 분사 → "LEVEL UP!" 픽셀 배너 슬라이드 인
  - 코인 획득: 동전이 HUD 메소 표시로 빨려 들어가는 모션
  - **프레임 원칙:** 모든 애니메이션은 픽셀 그리드 단위 이동(서브픽셀 금지)으로 도트 일관성 유지.

### 2-4. 반응형 전략

- **PC(메인, ≥1024px):** 3단 고정 레이아웃, 캘린더 풀사이즈.
- **태블릿(768~1023px):** 캘린더 + 퀘스트 2단, 친구 패널은 토글.
- **모바일(<768px):** 세로 스택. 상단 HUD 축소(레벨·EXP 바만), 캘린더는 주간 뷰 기본, 하단 고정 탭바(캘린더/퀘스트/친구/마이룸).
- **도트 깨짐 방지:** 에셋은 `@1x, @2x, @3x` 정수 배율 스프라이트로만 제공(소수 배율 스케일 금지), CSS `image-rendering: pixelated` 전역 적용, 레이아웃은 rem이 아닌 8의 배수 그리드로 설계.

## 3. 기술 스택 및 아키텍처

### 3-1. 기술 스택 추천

| 영역 | 선정 | 이유 |
|---|---|---|
| 프레임워크 | **Next.js 14 (App Router)** | SSR/CSR 혼합, 라우팅, Vercel 배포 최적 |
| 애니메이션 | **Framer Motion** + CSS sprite | 도트 이펙트·레벨업 연출 |
| 상태 관리 | **Zustand**(경량 전역) + **TanStack Query**(서버 상태) | 캐릭터/EXP는 Zustand, 일정 데이터 캐싱·동기화는 Query |
| 스타일 | **Tailwind CSS** + 커스텀 픽셀 토큰 | 8px 그리드 빠른 구현 |
| 백엔드 | **Supabase** (Postgres + Auth + Realtime + Storage) | 올인원 BaaS |
| 사운드 | **Howler.js** | 칩튠 BGM·효과음 제어 |

### 3-2. 데이터베이스 설계 (Supabase / Postgres)

```sql
-- 유저 프로필 (auth.users 확장)
profiles (
  id            uuid PK REFERENCES auth.users,
  nickname      text,
  level         int DEFAULT 1,
  exp           int DEFAULT 0,
  coins         int DEFAULT 0,
  title         text,          -- 현재 칭호
  avatar_config jsonb,         -- 캐릭터 외형/꾸미기
  streak_days   int DEFAULT 0,
  created_at    timestamptz DEFAULT now()
)

-- 일정(퀘스트)
schedules (
  id            uuid PK DEFAULT gen_random_uuid(),
  user_id       uuid REFERENCES profiles(id),
  title         text NOT NULL,
  memo          text,
  difficulty    int DEFAULT 1,        -- 1~3, EXP 산정용
  start_at      timestamptz,
  is_recurring  boolean DEFAULT false,
  recur_rule    text,                 -- RRULE 형식 (FREQ=WEEKLY;BYDAY=MO,WE)
  completed     boolean DEFAULT false,
  completed_at  timestamptz,
  created_at    timestamptz DEFAULT now()
)

-- 반복 일정의 개별 완료 이력 (반복은 인스턴스별 기록)
schedule_logs (
  id           uuid PK,
  schedule_id  uuid REFERENCES schedules(id),
  user_id      uuid REFERENCES profiles(id),
  target_date  date,
  completed     boolean,
  exp_gained   int
)

-- 친구 관계
friendships (
  id          uuid PK,
  requester   uuid REFERENCES profiles(id),
  addressee   uuid REFERENCES profiles(id),
  status      text DEFAULT 'pending',  -- pending/accepted/blocked
  created_at  timestamptz DEFAULT now()
)

-- 협동/경쟁 퀘스트
co_quests (
  id          uuid PK,
  title       text,
  goal_count  int,
  current_count int DEFAULT 0,
  ends_at     timestamptz
)
co_quest_members ( quest_id uuid, user_id uuid, contribution int )

-- 업적/배지
achievements ( id uuid PK, user_id uuid, badge_code text, earned_at timestamptz )

-- 아이템/인벤토리
items ( id uuid PK, name text, category text, price int, asset_url text )
inventory ( user_id uuid, item_id uuid, equipped boolean )
```

**설계 포인트:** 반복 일정은 `schedules`에 RRULE만 저장하고, **실제 완료 여부는 `schedule_logs`에 날짜별 인스턴스로 기록**해야 통계·EXP 중복 산정을 방지할 수 있습니다.

### 3-3. 기능 구현 로직

- **구글 OAuth:** Supabase Auth의 Google Provider 활성화 → `supabase.auth.signInWithOAuth({ provider: 'google' })`. 로그인 후 트리거(`on auth.users insert`)로 `profiles` 행 자동 생성.
- **일정 반복 로직:** **DB 쿼리 방식 권장**(cron보다 가벼움). 반복 일정은 RRULE로 저장하고, 클라이언트에서 조회 시점의 날짜 범위에 대해 `rrule.js`로 인스턴스를 동적 전개 → 해당 날짜에 `schedule_logs` 레코드가 없으면 '미완료'로 표시. 대량 알림이 필요하면 Supabase **pg_cron + Edge Function**으로 일 1회 인스턴스 프리젠레이트.
- **EXP 산정:** 완료 시 Edge Function 또는 Postgres **RPC 함수**로 처리(클라이언트 조작 방지). `exp_gain = base * difficulty` 계산 → `profiles.exp` 증가 → 레벨업 임계치 초과 시 `level++`, 잔여 EXP 이월. **반드시 서버 사이드에서** 트랜잭션 처리.
- **실시간 동기화:** Supabase **Realtime**으로 `co_quests`, `friendships` 테이블 구독 → 친구의 협동 퀘스트 기여나 응원이 즉시 반영. 개인 일정은 굳이 Realtime 불필요(TanStack Query 캐싱으로 충분).

### 3-4. 성능 최적화

- **이미지 스프라이트:** 캐릭터·이펙트·아이콘을 하나의 스프라이트 시트로 묶고 CSS `background-position`으로 출력 → HTTP 요청 최소화.
- **Lazy Loading:** 캘린더 밖(스크롤 영역, 친구 마이룸)의 에셋은 `next/image`의 `loading="lazy"`로 지연 로드.
- **에셋 포맷:** 도트는 PNG-8(인덱스 컬러)로 용량 최소화, 애니메이션은 스프라이트 시트 + CSS `steps()` (GIF/대용량 영상 지양).
- **렌더링:** 일정 100개+ 상황 대비 — 캘린더 셀은 가상화(react-window) 또는 월 단위 페이지네이션, 완료 항목은 별도 collapse.
- **캐싱:** Supabase Storage에 CDN 캐시 헤더 설정, 정적 도트 에셋은 `Cache-Control: immutable`.

경쾌·모험적·살짝 능청스러운 메이플 NPC 말투. **"~다!", "~라구!", "~할까?"** 같은 활기찬 어미, 과장된 감탄사("우와앗!", "두구두구..."), 모험·성장 메타포("모험가님", "여정", "퀘스트") 적극 사용. 사용자는 항상 **'모험가님'**으로 호칭.

## 5. 개발 및 검수 체크리스트

### 5-1. 기능 구현 체크리스트
- [ ] 구글 OAuth 로그인/로그아웃 정상 동작, 신규 가입 시 `profiles` 자동 생성 트리거 확인
- [ ] 일정 Supabase CRUD(생성·조회·수정·삭제) 전체 테스트
- [ ] 일회성 일정 — 단일 날짜 등록·완료 정상
- [ ] 반복 일정 — RRULE 전개 정확도(요일/주기), `schedule_logs` 날짜별 분리 기록 확인
- [ ] 친구 신청·수락·삭제 플로우, 협동 퀘스트 기여 합산 정확도
- [ ] EXP 산정 로직 — 난이도별 차등, 레벨업 임계치/이월 계산 오류 검증 (서버 RPC 단위 테스트)
- [ ] 코인·아이템 지급/차감 트랜잭션 무결성

### 5-2. 게이미피케이션 효과 검수
- [ ] 레벨업 시 효과음 출력 + 풀스크린 애니메이션 재생, 중복 트리거 방지
- [ ] 일정 완료 시 `+EXP` 모션·사운드 즉각 반응(0.3s 이내)
- [ ] 커스터마이징(아바타·테마) 설정 저장/불러오기 안정성, 새로고침 후 유지
- [ ] 사운드 on/off 토글 정상, 자동재생 차단 정책 준수
- [ ] 배지·칭호 획득 조건 정확 발동 및 알림

### 5-3. 보안 및 예외 처리
- [ ] 미로그인 상태에서 대시보드·API 접근 차단(라우트 가드 + 서버 검증)
- [ ] Supabase **RLS 정책** — 본인 데이터만 read/write(`auth.uid() = user_id`), 친구 데이터는 accepted 상태에서만 제한적 read
- [ ] EXP/코인은 클라이언트가 아닌 **서버 RPC로만 변경** 가능하도록 RLS 차단(조작 방지)
- [ ] 잘못된 일정 입력 예외 처리(빈 제목, 과거 날짜, 잘못된 RRULE)
- [ ] XSS 방지(일정 제목·메모 sanitize), API rate limiting

### 5-4. 테스트 케이스(부하·정합성)
- [ ] **렌더링 부하:** 일정 100·300·500개 등록 시 캘린더/퀘스트 목록 렌더 시간 측정(목표: 100개 기준 초기 렌더 <1s). 가상화 적용 전후 비교.
- [ ] **EXP 정합성:** 100개 일정을 순차/동시 완료 시 EXP 누락·중복 가산 없는지 검증 — 완료 후 `SUM(schedule_logs.exp_gained)`와 `profiles.exp` 일치 확인.
- [ ] **동시성:** 같은 일정을 빠르게 더블클릭 시 EXP 1회만 지급되는지(idempotency) 확인.
- [ ] **반복 일정 장기 테스트:** 1년치 주간 반복 일정 전개 시 성능·메모리 점검.
- [ ] **네트워크 예외:** 오프라인/지연 상황에서 완료 처리 롤백·재시도 동작.
- [ ] **크로스 디바이스:** PC↔모바일 동시 접속 시 Realtime 동기화 일관성.

## 🎯 전문가 조언 보강

조언하신 대로 **'디테일한 도트 감성 = 성공의 열쇠'**라는 진단에 전적으로 동의하며, 실행 우선순위를 제안합니다:

1. **에셋 파이프라인을 1순위로 확보** — Aseprite로 캐릭터 idle/완료/레벨업 스프라이트 시트를 먼저 제작하고, 정수 배율 출력 규칙을 초기에 못 박으세요. 도트는 후반에 끼워넣으면 일관성이 무너집니다.
2. **'완료 쾌감' MVP를 가장 먼저** — 로그인→일정 등록→완료→EXP→레벨업의 **단일 핵심 루프**를 Framer Motion으로 완성도 있게 만들고, 그 다음에 친구·상점을 붙이세요. 이 루프 하나의 손맛이 전체 앱 인상을 결정합니다.
3. **EXP/보상은 무조건 서버 사이드** — 클라이언트 계산은 조작·버그 위험이 큽니다. Edge Function/RPC로 일원화하세요.
